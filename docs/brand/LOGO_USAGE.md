# iGentic Logo Usage

Last updated: 2026-06-12

## Purpose

This file explains how the iGentic name and logo should be used by maintainers and contributors.

The project is open source, but the identity must stay clear, trustworthy and non-confusing.

## Allowed use

You may use the iGentic name and logo to:

- refer to the official open-source project,
- link to the repository,
- write tutorials, posts or reviews about the project,
- show compatibility with an unmodified upstream version,
- contribute design proposals in issues and pull requests,
- make screenshots or educational diagrams about the project.

## Not allowed without explicit maintainer approval

Do not use the iGentic name or logo to:

- imply official endorsement of a fork,
- sell a commercial product as official iGentic,
- publish an app that appears to be the official app,
- create confusingly similar branding,
- combine iGentic marks with Apple marks or other third-party marks,
- suggest Apple endorsement, partnership or certification,
- use the mark for unsafe automation, surveillance or credential-sharing products.

## Fork naming

Forks should make their status clear.

Good examples:

- `iGentic experimental fork by <name>`
- `<name>'s iGentic research branch`
- `iGentic-compatible policy experiment`

Avoid:

- `Official iGentic`
- `iGentic Pro`
- `iGentic App Store Edition`
- names that imply maintainer approval when none exists.

## Apple and third-party marks

Do not include Apple logos, Apple product silhouettes or Apple-owned interface imagery in official iGentic marks.

It is acceptable to describe the current research target as iPhone-focused when accurate, but the iGentic identity should stay independent and ownable.

## Asset quality rules

Official assets should:

- be committed as editable SVG when possible,
- include alt text guidance where used in docs,
- include monochrome variants,
- work on light and dark backgrounds,
- remain legible at small sizes,
- avoid embedded raster screenshots unless strictly necessary.

## Contributor proposals

Design proposals should be opened as issues or pull requests and include:

- goal,
- preview image or SVG,
- rationale,
- accessibility notes,
- trademark/confusion risk notes,
- license/source notes for any external assets.

## Legal note

This file is a practical project policy, not legal advice. It is intended to keep the open-source project clear, safe and non-confusing while allowing healthy community discussion and reuse.
